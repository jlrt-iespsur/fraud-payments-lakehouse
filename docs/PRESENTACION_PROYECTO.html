Slide 2 - Arquitectura

Arquitectura técnica y flujo de datos

El flujo separa claramente ingesta, persistencia cruda, enriquecimiento, detección, explotación analítica y análisis de relaciones.

1. Generador
Python publica eventos sintéticos de pago hacia Kafka con casos normales, reintentos y patrones sospechosos.
2. Kafka
Topic `payments` como capa de ingesta casi en tiempo real.
3. Spark Streaming
Consume Kafka y escribe Bronze en Iceberg sin transformar el dato funcional.
4. Iceberg + MinIO
Lakehouse con tablas Bronze, Silver, Gold sobre almacenamiento S3 compatible.
5. Trino
Consulta SQL unificada sobre Iceberg sin mover datos.
6. Superset
Dashboard de volumen, alertas y entidades de mayor riesgo.
7. Airflow + Neo4j
Compacta, exporta y carga el grafo para investigación conectada.
Qué destacar: Bronze conserva el dato original, Silver lo hace útil, Gold lo convierte en accionable y Neo4j permite pasar de una alerta aislada a una red sospechosa.

Slide 3 - Antes de arrancar

Requisitos previos y comprobaciones

Requisitos del entorno
  • Docker y Docker Compose instalados
  • Conexión a Internet en la primera ejecución para descargar paquetes de Spark
  • Puertos libres: `5432`, `7474`, `7687`, `8080`, `8081`, `8088`, `9000`, `9001`, `29092`
  • Espacio suficiente para imágenes y volúmenes locales
Comprobación mínima
docker --version docker compose version cp .env.example .env

La demo local puede ejecutarse con las credenciales por defecto definidas en `.env.example`.

Qué hay que vigilar Si `docker compose up` falla durante el build, lo primero es revisar conectividad de red y espacio en disco.
Qué conviene explicar El proyecto se ha planteado para ser reproducible, con contenedores aislados y una red común para todos los servicios.
Aclaración importante: Aunque aparezca `AWS_REGION` en la configuración, no se usa AWS real. MinIO habla API S3 y el cliente necesita una región ficticia para funcionar, por eso se fija `us-east-1`.
Kafka UI permite navegar por clusters, topics y mensajes. Para la demo interesa seguir la ruta `local -> Topics -> payments -> Messages`.
La Spark UI muestra la pestaña `Structured Streaming`. Ahí es donde conviene leer `Input Rate`, `Process Rate`, `Input Rows` y `Batch Duration`.

Slide 4 - Arranque

Cómo levantar toda la plataforma

./scripts/start_all.sh
Este script hace todo esto
  1. Crea directorios `runtime`.
  2. Construye las imágenes extendidas necesarias.
  3. Levanta PostgreSQL, MinIO, Kafka, Spark, Neo4j y el contenedor del generador.
  4. Inicializa el bucket de MinIO y el topic de Kafka.
  5. Inicializa Airflow y Superset.
  6. Levanta Trino, Airflow y Superset listos para su uso.
Qué debes verificar al terminar
  • MinIO responde en `http://localhost:9001`
  • Kafka UI responde en `http://localhost:8082`
  • Spark UI quedará visible en `http://localhost:4040-4050` al lanzar jobs Spark
  • Superset responde en `http://localhost:8088`
  • Airflow responde en `http://localhost:8081`
  • Neo4j responde en `http://localhost:7474`
  • Trino queda operativo en `http://localhost:8080`
Punto de atención: Si un servicio no responde, no sigas con la demo. Corrige primero la infraestructura, porque todo lo demás depende de esta base.
Un mal arranque arrastra errores al resto del pipeline 4 / 11
Slide 5 - Ingesta

Generación de datos y capa Bronze

Paso 1: dejar Bronze escuchando
./scripts/run_bronze.sh

Este job de Spark Structured Streaming escucha Kafka y escribe en `iceberg.payments.bronze_payments` sin alterar el esquema funcional del evento.

Paso 2: lanzar el generador
./scripts/run_generator.sh --events 5000 --sleep-ms 150

El generador simula pagos normales, reintentos, patrones sospechosos y algunos duplicados para poder demostrar la deduplicación en Silver.

Slide 6 - Transformación

Construcción de Silver y Gold

Silver
./scripts/run_silver.sh
  • tipado correcto
  • deduplicación
  • variables por ventanas temporales
Gold
./scripts/run_gold.sh
  • reglas de fraude
  • `fraud_alerts`
  • `graph_payments`
Qué hay que fijarse
  • Silver suele tener menos filas que Bronze por la deduplicación.
  • Gold contiene solo los casos sospechosos.
  • `reasons` explica por qué se dispara cada alerta.
  • Si Bronze sigue vivo, Silver y Gold suelen abrir nuevas Spark UI en puertos siguientes, por ejemplo `4041` y `4042`.
Mensaje para defender: Silver crea contexto. Gold convierte ese contexto en decisión. Esa separación hace la arquitectura más mantenible y más explicable.

Slide 7 - Validación SQL

Comprobación de resultados con Trino

Acceso al cliente SQL
docker compose exec trino trino
Consultas clave
SHOW TABLES FROM iceberg.payments; SELECT count(*) FROM iceberg.payments.bronze_payments; SELECT count(*) FROM iceberg.payments.silver_payments; SELECT count(*) FROM iceberg.payments.fraud_alerts; SELECT * FROM iceberg.payments.fraud_alerts ORDER BY risk_score DESC LIMIT 20;
Qué debes comprobar y explicar
  • Bronze refleja la ingesta original.
  • Silver filtra ruido y aporta contexto temporal.
  • Gold es la capa de acción y monitorización.
  • Las columnas `risk_score` y `reasons` hacen la alerta interpretable.

En paralelo puedes mantener abiertas Kafka UI y Spark UI para enseñar tanto la entrada de mensajes como su procesamiento.

Frase útil: No solo detectamos fraude; podemos justificar por qué una operación ha sido marcada como sospechosa.

Qué observar en cada visual

Slide 9 - Orquestación y grafo

Airflow y Neo4j para investigación conectada

Disparo del DAG

En Airflow, lanzar `fraud_graph_pipeline` con parámetros como estos:

{ "source_table": "graph_payments", "start_ts": "", "end_ts": "", "graph_name": "fraud_snapshot_demo" }

El DAG compacta Iceberg, exporta CSVs y los carga en Neo4j.

Consultas Cypher a enseñar
  • dispositivos compartidos por múltiples tarjetas
  • tarjetas utilizadas en múltiples países
  • comercios conectados a múltiples entidades sospechosas
  • agrupaciones de comportamiento anómalo
Qué aporta Neo4j: SQL detecta registros sospechosos. El grafo permite descubrir si varios casos aparentemente aislados forman parte de una misma red.

Slide 11 - Anexo de incidencias

Errores reales encontrados y cómo se resolvieron

El proyecto no solo se ha construido; también se ha endurecido durante la puesta en marcha. Este anexo resume los fallos reales más importantes y la corrección aplicada.

Incidencia
Impacto observado
Solución aplicada
Imagen pública de Spark no disponible
El arranque fallaba al depender de una tag externa inestable.
Se sustituyó por una imagen local propia en `docker/spark/Dockerfile`.
Airflow y Superset init mal encadenados
Los contenedores de inicialización no ejecutaban correctamente sus comandos.
Se reescribieron los `entrypoint` en `docker-compose.yml` para que la inicialización fuera determinista.
Iceberg contra MinIO sin región S3
Bronze fallaba con `Unable to load region` al crear tablas.
Se fijó `AWS_REGION=us-east-1` en Spark, scripts y Compose. No implica usar AWS real.
Bronze parecía arrancar, pero volvía al prompt
El job streaming quedaba ambiguo y costaba diagnosticar si seguía vivo.
Se mejoraron trazas, `trigger`, nombre de query y `docker compose exec -T`.
Trino no arrancaba o se caía
Errores de catálogo JDBC y reinicios por memoria con `Exited (137)`.
Se añadió `driver-class`, se ajustaron JVM y memoria, y se estabilizó el servicio.
Gold fallaba en PySpark
`array_remove` sobre columnas rompía el job y después hubo problemas de JVM.
Se rehizo la construcción de razones y se simplificó la escritura de Iceberg.
Neo4j y Airflow muy sensibles con 8 GB de RAM
Servicios caídos, DAGs rotos o Browser inestable durante la demo.
Se bajaron recursos, se documentaron apagados selectivos y se operó por fases.
Compactación de Iceberg bloqueando el DAG
El paso `compact_iceberg_table` podía cortar el pipeline de grafo.
La compactación se dejó como best-effort para no impedir la exportación y la carga en Neo4j.
Mensaje para defensa: El valor del proyecto no está solo en el resultado final, sino en haber identificado problemas reales de integración entre Spark, Iceberg, Trino, Airflow, MinIO y Neo4j, y haber dejado el sistema operativo y documentado.
